OUR USE CASE

We're building an AI-powered call coaching platform that automatically pulls call recordings from BatchDialer, transcribes them, and grades them against custom rubrics. We poll the /api/cdrs endpoint every 30 minutes to pick up new calls.


EXACT REQUEST FORMAT

Endpoint: GET https://app.batchdialer.com/api/cdrs

Headers:
- X-ApiKey: [our API key]
- Accept: application/json

Query Parameters (previous setup that caused timeouts):
- callDate: ISO 8601 timestamp (e.g., 2026-02-10T02:00:00.000Z) — start of the time range
- page: 1, 2, 3... — paginated through results
- pagelength: 100 — records per page

The callDate parameter was set to 2 hours before the current time. We were NOT using a callDateEnd parameter — the request was open-ended from the start time to "now."


WHAT WE'VE CHANGED (BASED ON YOUR SUGGESTIONS)

We've already implemented the following improvements:

1. Smaller Time Windows (15-minute chunks)
Instead of requesting the full 2-hour range in one call, we now break the range into 15-minute windows. Each window gets its own paginated request sequence. For example, a 30-minute sync now makes 2 windowed requests instead of 1 large one.

2. Reduced Page Size
Dropped from pagelength=100 to pagelength=50 to reduce per-request payload.

3. Exponential Backoff Retry
If a request times out or returns a 5xx error, we retry up to 3 times with exponential backoff (2s, 4s, 8s + jitter). We fail gracefully per-window — if one 15-minute window fails after retries, we log the error and continue with the next window.

4. Shorter Client Timeout
Reduced our client-side timeout from 120 seconds to 30 seconds so we fail fast and retry, rather than waiting for the server to close the connection.

5. Reduced Default Lookback
Changed the default lookback from 2 hours to 30 minutes for first-time syncs.
